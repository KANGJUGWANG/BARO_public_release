# BARO 프로젝트 의사결정 여정

## 1. 왜 항공권 가격 추천 서비스인가

항공권 검색 서비스는 이미 많지만, 사용자가 실제로 어려워하는 지점은 “어디서 검색할 것인가”보다 “지금 사도 되는가”에 가깝다. 같은 노선과 날짜라도 가격은 시간에 따라 변하고, 사용자는 현재 가격이 싼지 비싼지 판단하기 어렵다. BARO는 이 문제를 가격 관측 이력과 현재 후보 가격을 함께 분석해 구매 시점 판단을 보조하는 서비스로 정의했다.

## 2. 사용자 문제 정의

사용자가 겪는 핵심 문제는 다음과 같다.

- 여러 항공권 후보 중 어떤 가격이 상대적으로 좋은지 즉시 판단하기 어렵다.
- 기다리면 싸질지, 지금 사는 편이 나을지 알기 어렵다.
- 단순 최저가 목록은 가격의 시간적 맥락을 설명하지 않는다.
- 왕복은 출국편과 귀국편 조합까지 고려해야 하므로 판단 부담이 더 크다.

따라서 서비스 목표는 “최저가 검색”이 아니라 “현재 가격 기준 구매 의사결정 보조”로 잡았다.

## 3. 왜 BUY/WAIT인가

복잡한 회귀값이나 내부 모델 점수를 바로 보여주면 사용자가 행동으로 연결하기 어렵다. 그래서 BARO는 추천 결과를 BUY와 WAIT으로 단순화했다.

- BUY: 현재 가격 기준 구매를 권장한다.
- WAIT: 추가 하락 가능성을 고려해 대기를 권장한다.

이 표현은 가격을 보장한다는 뜻이 아니라, 현재 관측 이력과 모델 출력에 근거한 행동 보조 신호다.

## 4. 왜 머신러닝 접근인가

항공권 가격은 노선, 항공사, 항공편, 출발일까지 남은 기간, 최근 가격 흐름, 누적 최저/최고/평균 대비 위치가 함께 작용한다. 단순 규칙만으로는 가격 하락 가능성을 안정적으로 판단하기 어렵다. 그래서 BARO는 과거 관측 가격 이력을 feature로 만들고, 미래 가격 하락 가능성을 모델로 추정하는 구조를 채택했다.

## 5. 초기 라벨 정의

초기 라벨은 미래 최저 가격 기준 절약 가능성이 해당 항공권의 변동성 기준보다 큰지를 사용했다. 요약하면 다음과 같다.

- 미래 가격이 충분히 내려갈 가능성이 크면 WAIT.
- 그렇지 않으면 BUY.
- 변동성이 큰 항공편은 더 큰 하락을 요구하고, 변동성이 작은 항공편은 작은 하락도 의미 있게 본다.

이 방식은 항공편별 가격 변동성을 반영한다는 장점이 있었지만, 사용자가 체감하는 “기다릴 가치”와 항상 일치하지는 않았다.

## 6. 기존 기준에서 발견한 문제

Phase 12~14 audit에서 중요한 문제는 BUY 비율 자체가 아니라 BUY regret이었다. 즉 BUY라고 했는데 72시간 안에 체감 가능한 가격 하락이 발생하는 경우가 서비스 품질 문제로 정의됐다.

또한 confidence라는 표현이 실제 검증 정확도가 아니라 모델 판단 강도에 가까웠다. 이 때문에 “신뢰도 99%”처럼 보이는 UI는 사용자가 모델 정확도로 오해할 수 있었다.

## 7. 사용자 체감 기준으로의 변화

프로젝트는 내부 라벨의 수학적 일관성보다 사용자가 체감하는 결과를 더 중시하는 방향으로 평가 기준을 바꿨다. 핵심 질문은 다음으로 바뀌었다.

- BUY 추천 후 가까운 시간 안에 의미 있는 가격 하락이 있었는가?
- WAIT 추천 후 실제로 기다릴 가치가 있는 가격 하락이 있었는가?

이 흐름에서 72시간, 20,000원, 3% 기준이 주요 실험 기준으로 채택됐다.

## 8. 왜 72시간, 20,000원, 3%인가

72시간은 사용자가 추천 이후 실제로 기다릴 수 있는 현실적인 시간 범위다. 20,000원은 항공권 구매에서 체감 가능한 절대 금액 기준이고, 3%는 가격대가 다른 항공권 간 비교를 위한 상대 기준이다. 둘 중 하나만 쓰면 왜곡이 생길 수 있어 “20,000원 이상이면서 3% 이상”이라는 조합을 사용했다.

이 기준은 모델 내부 목적 함수라기보다 서비스 품질 평가와 최종 라벨 후보를 설명하기 위한 사용자 체감 기준으로 사용됐다.

## 9. 왜 Stage1/Stage2 구조인가

BARO의 모델은 2단계 구조를 유지했다.

1. Stage1: 현재 후보와 과거 이력 기반으로 예상 절약 가능성을 추정한다.
2. Stage2: Stage1 출력과 원래 feature를 함께 사용해 WAIT 확률을 산출한다.

이 구조는 “얼마나 내려갈 수 있는가”와 “대기할 만한가”를 분리한다. Stage1 값은 Stage2의 보조 feature로 들어가며, 최종 decision은 Stage2의 WAIT 확률과 threshold로 결정된다.

## 10. 왜 편도와 왕복을 분리했는가

편도와 왕복은 데이터 구조와 판단 난도가 다르다.

- 편도는 단일 항공편 가격 흐름 중심이다.
- 왕복은 출국편과 귀국편 조합, 귀국일, 체류일수, 귀국편 번호까지 고려한다.
- 왕복 후보 수와 조합 변동성이 더 크다.

따라서 모델 버전과 threshold를 분리해 운영한다. 최종 live 기준은 편도 threshold 0.80, 왕복 threshold 0.65로 정리됐다.

## 11. 최종 학습 데이터 범위

최종 재학습 문서 기준 공개 가능한 요약 범위는 다음과 같다.

- 수집 기간: 2026-04-16 08:00부터 2026-06-04 00:00까지
- 관측 시점: 147개
- 서비스 DB 관측 수: 141,120건
- 학습 가능 항공권 row: 4,562,741건
- 편도 학습 row: 989,669건
- 왕복 학습 row: 3,573,072건

원천 데이터와 모델 바이너리는 공개 저장소에 포함하지 않는다.

## 12. 최종 모델 운영 반영

최종 모델은 Stage1과 Stage2를 모두 재학습한 full retrain 후보를 runtime smoke로 검증한 뒤 live backend에 반영됐다. 운영 확인에서 편도와 왕복 각각의 model_version과 threshold가 분리되어 확인됐다.

- 편도 model_version: finaltest_expc_full_final_v1_thr065_oneway
- 왕복 model_version: finaltest_expc_full_final_v1_thr065_roundtrip
- 편도 threshold: 0.80
- 왕복 threshold: 0.65

## 13. 왜 UI를 가격 하락 기대 강도로 바꿨는가

기존 confidence는 정확도가 아니라 WAIT 확률 또는 그 보수에 가까운 판단 강도였다. 그래서 “신뢰도”라는 표현은 제거하고, 사용자가 실제로 이해할 수 있는 “가격 하락 기대 강도”로 바꿨다.

이 변경은 모델 decision rule을 바꾼 것이 아니라, 같은 모델 출력을 더 정확하게 설명하기 위한 UI 표현 변경이다.

## 14. 공개와 비공개의 경계

공개 가능한 것은 문제 정의, 모델 구조, 라벨 철학, 평가 기준, 요약 수치, API 계약, 화면 동작 설명이다. 공개하지 않는 것은 운영 서버 주소, 개인 경로, 비공개 설정값, 모델 바이너리, 원천 DB, 대용량 학습 데이터, 실험 산출물 원본이다.

BARO 공개 문서는 재현 가능한 구조와 의사결정 근거를 제공하되, 운영 환경과 민감 자산은 분리한다.
