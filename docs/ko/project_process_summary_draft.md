# BARO 프로젝트 진행 과정 요약 초안

## 1. 문제 선정

BARO는 항공권 검색 자체보다 구매 시점 판단을 돕는 문제에 집중했다. 사용자는 최저가 목록을 볼 수 있어도 현재 가격이 앞으로 더 내려갈지 알기 어렵다.

## 2. 데이터 수집과 관측 이력 구축

대상 노선과 날짜 범위를 정하고 반복 관측을 통해 가격 이력을 축적했다. 편도와 왕복은 후보 구조가 다르므로 이후 모델과 API contract도 분리했다.

## 3. 초기 모델 설계

초기 모델은 2단계 구조로 설계했다.

- Stage1: 미래 절약 가능성 추정
- Stage2: WAIT 확률 산출

최종 결정은 Stage2의 WAIT 확률과 threshold 비교로 이뤄진다.

## 4. 문제 발견

운영 관찰과 audit에서 BUY 추천이 많고, 기존 confidence 표시가 실제 정확도로 오해될 수 있다는 문제가 확인됐다. 핵심 개선 목표는 BUY 비율 자체가 아니라 BUY regret 감소로 재정의했다.

## 5. 라벨/평가 기준 재정립

사용자가 체감할 수 있는 기준으로 72시간 안에 20,000원 이상이면서 3% 이상 가격이 내려가는지를 주요 평가 기준으로 검토했다. 이 기준은 Exp-C로 정리됐고, 실험을 통해 기존 기준보다 서비스 지표가 나은 후보로 남았다.

## 6. 최종 재학습

최종 데이터 freeze 이후 Stage1과 Stage2를 다시 학습했다. 편도는 편도 전용 모델, 왕복은 왕복 전용 모델로 분리했고 threshold도 trip별로 선택했다.

## 7. 운영 반영

runtime smoke와 model-info 확인을 거쳐 최종 모델을 live backend에 반영했다. UI에서는 confidence 표현을 제거하고 가격 하락 기대 강도와 WAIT 기준선을 표시하도록 정리했다.

## 8. 공개 저장소 정리

공개 저장소는 소스, 공개 문서, 예시 환경 파일 중심으로 구성한다. 원천 데이터, 모델 바이너리, 운영 설정, 실험 raw output은 포함하지 않는다.
